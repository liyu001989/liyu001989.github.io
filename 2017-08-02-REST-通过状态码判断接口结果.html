2017-08-02 为什么是这样的接口

既然是决定写 REST 风格的接口，根据不同的状态码来判断接口的返回接口，应该是特别正常的一件事情，但是，当客户端的同学们，过来跟你理论的时候，还是需要一些功夫才能说服他们的。

接口的返回，大致上有两大阵营

  1. 所有的接口返回的状态码为200，body 中一定存在 message，code 用来说明返回结果。然后还有可能是请求成功了，直接返回数据，失败了，会存在 errorcode，及 message。
  2. REST, 先判断状态码，body 中只有资源数据。

国内绝大部分的接口都是第一种，国外写 REST 风格的还算多一些。所有人都习惯了第一种方式，所以就会有下面这些问题

客户端的疑问

  • ios, nfnetworking 默认只有 200 才算成功，否则会抛异常，需要单独处理。
  • http 请求是网络层的事情，真正的成功失败是业务上的事情。为什么要把两件事情混在一起。
  • 没有用过 put，patch 和 delete。
  • 需要额外的学习成本。

其实客户端的同学们是一定会有各种各样的疑问和不情愿的，谁不想快点搞定回家休息啊，画个界面已经够烦的了，各种各样的需求，动不动就改界面，本来就忙的不行。没有人回去关心，你是个什么风格的接口，接口写成这样有啥好处，大家的出发点只有一个，我不方便处理。

从心态上，情感上，我还是理解客户端的，能不能顺利的说服大家，接受这样的接口取决于你处在什么样的职位上，你有什么样的领导，你的专业素养能不能让大家信服，其他部门的领导愿不愿意做出改变，工期上允不允许等等等等。
